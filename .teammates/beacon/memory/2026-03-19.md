## Task: Review Multi-Human Collaboration Spec

Scribe created the design spec at `.teammates/scribe/docs/specs/F-multi-human-collaboration.md` and handed off a technical feasibility review to Beacon. Reviewed the full spec against the current CLI/orchestrator architecture.

### Key findings

**Phase 1 (file-based) is viable and the right starting point:**
- `_handoffs/` directory follows the existing folder naming convention (`_` prefix = shared non-teammate, checked in)
- The CLI already has handoff parsing (`findHandoffBlocks` in cli-proxy.ts) and approval UI (`renderHandoffs` in cli.ts) — extending these to write/read file-based handoffs is straightforward
- No server needed — git push/pull is the transport layer
- Biggest risk: race conditions if two humans push handoff files simultaneously (git handles this via merge, but JSON files don't merge cleanly — consider one file per handoff)

**Server architecture (Phase 2) — feasible but needs scoping:**
- WebSocket presence is standard — 60s heartbeat, 3-miss timeout is fine
- GitHub OAuth maps cleanly to avatar folders (GitHub alias = folder name)
- REST API surface is small and well-defined
- The server is a NEW package (`@teammates/server`) — significant new surface area
- Question: does the server run agents? Or do agents always run on the client? The spec says "AI teammates execute via the server" — this has major implications for compute, API keys, and cost

**Client connection model — compatible with current architecture:**
- `config.json` with `server` field is clean — doesn't conflict with `settings.json`
- Queue digest on connect maps to a new startup phase (after `init()`, before REPL)
- WebSocket can be a background connection managed by a new module — doesn't touch the REPL loop
- New commands (`/accept`, `/delegate`, `/reply`, `/dismiss`) are standard SlashCommand registrations

**Architecture conflicts — minor, all resolvable:**
1. `TeammateConfig` needs a `type: "human" | "ai"` field — Registry must parse this from SOUL.md
2. `QueueEntry` needs a `"handoff-queue"` variant for file-based handoff processing
3. `HandoffEnvelope` is close to the spec's handoff JSON but missing `priority`, `expires`, `status`, `thread_id` — needs extension
4. `/status` command collision — currently shows teammate statuses, spec adds presence info. Can merge (show both)
5. Orchestrator's `assign()` needs a gate: if target is a human avatar and human is offline, queue instead of execute

**Concerns:**
- "AI teammates execute via the server" — this is the biggest architectural question. Currently ALL execution is client-side. Moving AI execution to the server means the server needs agent binaries, API keys, and compute. Recommend keeping AI execution client-side in Phase 2 and only moving it server-side in a later phase.
- Handoff threads add complexity — a thread is essentially a conversation, which means state management, ordering, and a new UI paradigm. Consider shipping threads as a Phase 2.5 feature, not Phase 2.
- The `@github` synthetic teammate is elegant but needs a webhook receiver on the server side — adds deployment complexity.

### Files changed
- None (review only)

## Task: Implement user avatar system (Campfire Phase 1)

3. **`registry.ts`** — Added `parseType()` that looks for `**Type:** human` in SOUL.md. Defaults to "ai".

4. **`cli.ts`** — Major changes:
   - Interview now collects `alias` as first question (required)
   - `createUserAvatar()` creates `.teammates/<alias>/` with SOUL.md (`**Type:** human`) and WISDOM.md
   - `readUserAlias()` reads alias from USER.md
   - `registerUserAvatar()` registers the avatar with presence "online" and type "human"
   - `selfName` getter returns `userAlias ?? adapterName`
   - `/status` shows presence indicators (● green/red/yellow) for all teammates
   - User avatar shown first with "(you)" label; coding agent hidden
   - `@everyone` excludes both avatar and adapter
   - Roster excludes adapter (internal)

5. **`template/USER.md`** — Added Alias section as first field

6. **Tests** — All 162 CLI tests pass. Added `type: "ai"` to all test fixtures.

### Key decisions
- Adapter stays registered for internal tasks (summarize, btw, debug) but hidden from user-facing surfaces
- Human avatar's SOUL.md has `**Type:** human` marker — parsed by Registry
- Local user is always "online"; remote humans (future) default to "offline"
- `selfName` used for identity/display, `adapterName` used for execution
- Alias normalized: lowercase, only `[a-z0-9_-]` characters
