## Task: Add process diagnostics for empty agent responses

User reported occasional "no response text" with empty debug output when running teammate tasks via `claude -p`. The `/debug` view showed Duration: 143.8s, Status: OK, but completely empty raw output ŌĆö zero diagnostic info.

### Root cause analysis
`spawnAndProxy` had three diagnostic blind spots:
1. **Exit code discarded** ŌĆö `child.on("close", (_code) => ...)` ignored the exit code
2. **stdout/stderr merged** ŌĆö both went into one `captured` array, so we couldn't tell if stderr had error/tool-call output when stdout was empty
3. **No agent debug logs** ŌĆö Claude CLI supports `--debug-file` for internal debug logs, but we weren't using it

### What was done
1. **`SpawnResult` type** (cli-proxy.ts) ŌĆö New structured return with separate `stdout`, `stderr`, `exitCode`, `signal`, `timedOut`, `debugFile`
2. **Separate stream capture** ŌĆö stdout/stderr now in separate buffers; `close` handler captures both `code` and `signal`
3. **`--debug-file` flag** ŌĆö Claude preset now generates a temp file path and passes `--debug-file <path>` to capture Claude Code's internal debug log
4. **`TaskResult.diagnostics`** (types.ts) ŌĆö New optional field carrying all process metadata through the result chain
5. **`writeDebugEntry`** (cli.ts) ŌĆö Now logs: exit code, signal, timeout, debug file path, and stderr in a separate `<details>` block
6. **Empty response UI** (cli.ts) ŌĆö Shows exit code/signal warnings and debug file path inline below the "no response text" message

### Key decisions
- `supportsDebugFile` is a preset flag ŌĆö only claude gets `--debug-file`, other presets (codex, aider) are unchanged
- Debug file uses `tmpdir()` and includes teammate name + timestamp for easy identification
- stderr is always captured separately but still included in the combined `output` for backward compatibility
- Diagnostics are optional on TaskResult so existing code (tests, echo adapter) doesn't need changes

## Task: Ctrl+click file paths + default routing to active teammate

Two UX improvements per user request.

### 1. Ctrl+click absolute file paths in markdown
Extended ChatView's ctrl+click handler to detect absolute file paths alongside URLs. Paths matching `C:\...` (Windows) or `/...` (Unix) are now clickable. File paths emit a `"file"` event (separate from `"link"`) so the CLI can distinguish them. CLI handles both events with the platform's default opener (`open`/`start`/`xdg-open`).

**Key decisions:**
- `FILE_PATH_REGEX` matches Windows drive letters (`C:\`) and Unix absolute paths (`/`)
- When a line has mixed URLs and file paths, position-based hit testing determines which one was clicked
- `"file"` event is separate from `"link"` to allow different handling in the future

### 2. Default routing to active teammate
When the user sends a message without `@mentions`, the CLI now defaults to the teammate from the last response (`lastResult.teammate`) instead of immediately running `orchestrator.route()`. This makes replies naturally continue the conversation with whoever just responded. Auto-routing only kicks in when there's no prior conversation context.

**Key decisions:**
- Priority: `lastResult.teammate` ŌåÆ `orchestrator.route()` ŌåÆ `adapterName`
- This only affects un-mentioned messages ŌĆö explicit `@mentions` still override
