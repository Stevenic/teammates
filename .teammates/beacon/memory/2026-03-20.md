## Task: Reorder startup onboarding — user first, then team

User requested that the startup flow always onboard the user first (if USER.md is missing), then do team onboarding only if .teammates/ was missing.

### What was done

1. **`start()` in cli.ts** — Restructured the startup lifecycle:
   - Detect if `.teammates/` is missing (`isNewProject` flag)
   - If missing, create the directory immediately and show a welcome logo
   - Always check for missing USER.md → run `runUserSetup()` first
   - Only then, if `isNewProject`, show team onboarding menu via `promptTeamOnboarding()`

2. **`promptOnboarding()` → `promptTeamOnboarding()`** — Renamed and simplified:
   - No longer creates `.teammates/` (already done by `start()`)
   - Returns `boolean` (continue/exit) instead of `string | null` (path)
   - Takes `teammatesDir` as a parameter instead of building it internally
   - Removed logo display (moved to `start()`)
   - Solo mode no longer creates the directory or shows the "Created" message

### New startup flow
```
start()
  → findTeammatesDir()
  → if missing: create .teammates/, show welcome logo
  → if USER.md missing: runUserSetup() (GitHub / Manual / Skip)
  → if isNewProject: promptTeamOnboarding() (New team / Import / Solo / Exit)
  → init orchestrator
```

### Key decisions
- User onboarding always happens before team onboarding — user identity is needed first
- `.teammates/` is created unconditionally if missing — both user and team setup need it
- Existing projects with `.teammates/` but missing USER.md still get user setup (e.g., after git clone)
- Team onboarding menu is unchanged (same 4 options)

## Weighed in on Hands project — Beacon's scope

Responded to @everyone asking what each teammate needs to do for P2 Hands (cross-agent computer use).

### Beacon's deliverables

**1. MCP Server Lifecycle in CLI (~200 LOC)**
- CLI needs to spawn/manage the Hands MCP server as a child process during sessions
- Start on demand (first computer use task) or eagerly based on config
- Shutdown on session exit, crash recovery with restart
- Health check endpoint so adapters can verify the server is alive before routing

**2. Adapter Preset Capabilities for Computer Use (~100 LOC)**
- Extend `AgentPreset` with a `capabilities.computerUse` field (ties into AI1 goal)
- Three tiers: `"native"` (agent has built-in computer use → passthrough), `"mcp"` (agent supports MCP → inject Hands server config), `"none"` (graceful degradation → skip or warn)
- Claude: native passthrough via `--computer-use` flag
- Codex: MCP tier via `--mcp-server` flag pointing to Hands server
- Aider: likely `"none"` tier initially

**3. MCP Config Injection in cli-proxy.ts (~80 LOC)**
- When a task targets a teammate with computer use enabled, inject the Hands MCP server config into the agent spawn
- For Claude: generate a temp `mcp_config.json` with the Hands server entry, pass via `--mcp-config`
- For Codex: pass `--mcp-server` with the server URL
- Cleanup temp config files in `finally` block

**4. `/configure hands` Service Integration (~60 LOC)**
- Add Hands to the service detection system (like GitHub)
- Detect if the MCP server binary is available
- Show install instructions if missing
- Banner shows Hands status (configured/not configured/missing)

**5. Recall: Visual Context Indexing (~150 LOC, stretch)**
- If Hands captures screenshots as part of task execution, recall could index them
- Embeddings for image descriptions (OCR text or agent-generated captions stored as metadata)
- This is a stretch goal — depends on what metadata the MCP server exposes

### Dependencies
- **AI1 (Preset Capabilities Declaration)** — must land first; Hands needs the `capabilities` field on `AgentPreset`
- **EP1 (MCP Integration)** from goals — Hands is effectively a specific use case of EP1
- **Scribe's spec** — need the MCP tool schema (screenshot, click, type, scroll) before implementation

### Key decisions
- EP1 (MCP Integration) is the prerequisite infrastructure — Hands is the first consumer
- No new packages needed; MCP server management lives in `packages/cli/src/`
- Computer use capability is opt-in per teammate via SOUL.md or config
