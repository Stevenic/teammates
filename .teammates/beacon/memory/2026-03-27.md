## Task: Implement interrupt-and-resume (Phase 1)

Implemented the `/interrupt <teammate> [message]` command per Scribe's spec at `.teammates/scribe/docs/specs/F-interrupt-and-resume.md`. This allows users to kill a running agent mid-task and resume with context — the agent sees its own prior work and the user's steering message.

### What was done

**1. Types (`types.ts`):**
- Added `InterruptState` interface — captures teammate, original task/prompt, conversation log, elapsed time, tool call count, and files changed

**2. Adapter interface (`adapter.ts`):**
- Added optional `killAgent?(teammate): Promise<SpawnResult | null>` to `AgentAdapter`

**3. CliProxyAdapter (`cli-proxy.ts`):**
- Added `activeProcesses` map tracking `{ child, done, debugFile }` per teammate
- Refactored `spawnAndProxy` from inline Promise constructor to deferred pattern — `done` promise is shared between `executeTask` (normal flow) and `killAgent` (interrupt flow)
- Child processes registered on spawn, removed on close
- `killAgent()` sends SIGTERM → 5s → SIGKILL, then awaits the shared `done` promise

**4. Log parser (`log-parser.ts` — new):**
- `parseClaudeDebugLog()` — structured parsing of Claude `--debug-file` output
- `parseCodexOutput()` — JSONL parsing of Codex `--json` stdout
- `parseRawOutput()` — fallback pattern matching for any agent's stdout
- `formatLogTimeline()` — groups consecutive same-action entries, numbers steps
- `buildConversationLog()` — orchestrates parsing with token budget truncation

**5. `/interrupt` command (`cli.ts`):**
- New slash command: `/interrupt <teammate> [message]` (alias: `/int`)
- Validates teammate has active task, adapter supports kill
- Kills agent, captures SpawnResult, parses conversation log
- Builds resume prompt with `<RESUME_CONTEXT>` section
- Cleans up active task state and queues the resumed task
- `buildResumePrompt()` assembles: what was done + interruption message + original task

**6. Orchestrator (`orchestrator.ts`):**
- Added `getAdapter()` public method for `/interrupt` to access the adapter

**7. Exports (`index.ts`):**
- Exported all new log-parser functions and `InterruptState` type

### Key decisions
- `spawnAndProxy` uses deferred promise pattern — both `executeTask` (awaits normally) and `killAgent` (kills then awaits) share the same `done` promise
- Log parser extracts file paths and search queries, NOT full file contents — keeps resume prompt compact
- `formatLogTimeline` groups 4+ consecutive same-action entries to collapse bulk writes (e.g., "Write 42 items: a.ts, b.ts, c.ts (+39 more)")
- Resume prompt goes through normal `buildTeammatePrompt` wrapping (not raw) so the agent has full identity/memory context
- Default steering message: "Wrap up your current work and report what you've done so far."

## Task: Version bump to 0.6.0 + migration logic

Bumped version to 0.6.0 and added migration logic that compresses all historical daily logs and re-indexes on first startup when the project's .teammates version is missing or < 0.6.0.

### What was done

**1. Version bump:** All 3 package.json files (cli, recall, consolonia) bumped 0.5.3 → 0.6.0, plus dependency refs in cli/package.json.

**2. New functions in `compact.ts`:**
- `findUncompressedDailies(teammateDir)` — scans memory dir, returns dates of daily logs without `compressed: true` frontmatter
- `buildMigrationCompressionPrompt(teammateDir, teammateName, dailies)` — builds a bulk compression prompt for an agent to compress multiple daily logs in one pass

**3. Migration logic in `cli.ts`:**
- `semverLessThan(a, b)` — static helper for semver comparison
- In `startupMaintenance()`, captures `checkVersionUpdate()` return value
- If previous version is missing or < 0.6.0: finds all uncompressed dailies per teammate, queues system tasks with `migration: true` flag
- `pendingMigrationSyncs` counter tracks completion; when all migration tasks finish, triggers full `syncRecallIndex` and shows success message

**4. Types (`types.ts`):** Added `migration?: boolean` to agent QueueEntry variant.

**5. Exports (`index.ts`):** Exported `findUncompressedDailies` and `buildMigrationCompressionPrompt`.

### Key decisions
- Migration uses LLM-based compression (system tasks) for quality — same rules as daily compression
- One system task per teammate with all their uncompressed logs listed — agent reads/compresses each file
- Re-index fires only after ALL migration tasks complete (counter-based tracking)
- `semverLessThan` is a generic utility — can be reused for future migrations

## Task: Attention dilution fixes + daily compression + version tracking

Implemented Lexicon's 5 proposed fixes for the attention dilution problem (Scribe spending all tool calls on housekeeping instead of processing the task), plus Steve's additions: daily log compression and version tracking.

### What was done

**Fix 1 — Deduplicate recall against daily logs (`adapter.ts`):**
- Before injecting recall results, filter out any result whose URI matches a daily log date already present in `<DAILY_LOGS>`
- Prevents recall from re-injecting content the prompt already contains

**Fix 2 — Cut daily log budget (`adapter.ts`):**
- Reduced `DAILY_LOG_BUDGET_TOKENS` from 24K to 12K
- Freed budget flows to recall via the existing spillover mechanism
- Updated 3 tests in `adapter.test.ts` to match new budget

**Fix 3 — Echo user's request at bottom edge (`adapter.ts`):**
- Extracts user's actual message from `taskPrompt` (after last `---` separator)
- If <500 chars, echoes verbatim as `**THE USER'S REQUEST:**` at bottom of `<INSTRUCTIONS>`
- If ≥500 chars, adds a pointer: "The user's actual request is at the end of `<TASK>`"

**Fix 4 — Task-first priority statement (`adapter.ts`):**
- Added bold priority statement at top of `<INSTRUCTIONS>`: task first, housekeeping after

**Fix 5 — Conversation history as file reference (`cli.ts`):**
- When conversation context exceeds 8K chars, writes to `.teammates/.tmp/conversation.md`
- Prompt gets a pointer to the file instead of inline content
- Reduces prompt noise for long conversations

**Daily log compression (`compact.ts` + `cli.ts`):**
- `buildDailyCompressionPrompt()` checks if yesterday's log needs compression (new day boundary, not already compressed)
- Queued as system task during startup maintenance
- Compressed logs marked with `compressed: true` frontmatter to prevent re-compression
- Keeps task headers + one-line summaries + key decisions + file lists (3-5 lines per task)

**Version tracking (`cli.ts`):**
- `checkVersionUpdate()` reads/writes `cliVersion` field in `.teammates/settings.json`
- Compares with `PKG_VERSION` on startup
- On major/minor version change, shows update notification in feed
- Foundation for future auto-migration on version bumps

### Key decisions
- Daily log budget halved (24K→12K) because past logs are reference material, not active context
- Conversation file threshold (8K chars) balances inline convenience vs. noise reduction
- Daily compression uses system task (LLM-based summarization) for quality compression
- Version comparison uses major.minor only — patch bumps don't trigger notifications
