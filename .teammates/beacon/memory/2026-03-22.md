## Task: Fix broken handoffs — agents not using fenced block format

User reported that multiple teammates (Scribe, Lexicon) claim to queue handoffs but nothing actually happens. Investigated debug logs.

### Root cause
Agents are NOT using the ```handoff fenced code block format. They write about handoffs in plain English instead:
- Scribe: "And yes — please kick off the Beacon handoff whenever you're ready" (no block)
- Scribe: "Handoff to Beacon is done." (no block)
- Lexicon: "All done. 18 files updated, 2 handoffs queued" (no block)

The `findHandoffBlocks()` parser requires exact `/```handoff\s*\n@(\w+)\s*\n/` format — correctly finds nothing because there's nothing to find.

### Why agents don't follow the format
The handoff instruction was buried as one paragraph inside a 20-line Output Protocol section. The format example used nested backticks (code fence inside code fence) which was ambiguous.

### What was done

**Fix 1 — Separated handoff protocol into its own section (adapter.ts):**
- Pulled handoff instructions out of Output Protocol into a new `## Handoffs (IMPORTANT)` section
- Used 4-space indentation for the format example instead of nested code fences — unambiguous rendering
- Added explicit warning: "Mentioning a handoff in plain English does NOT work"
- Added "Do NOT just say 'I'll hand this off' in prose — that does nothing"

**Fix 2 — Natural-language handoff fallback (cli-proxy.ts):**
- Added `findNaturalLanguageHandoffs()` as a fallback when no fenced blocks are found
- Detects patterns: "hand off to @name", "handing this to @name", "delegate to @name", "queued handoff to @name"
- Only matches known teammate names (from `teammateNames` list)
- Extracts surrounding text as the task description
- Deduplicates by target name
- Only fires when zero fenced blocks are found (fenced blocks are always preferred)

### Two layers of defense
1. **Better prompt** — agents should follow the fenced block format now that it's clear
2. **Fallback parser** — catches the common failure mode when agents still use natural language

## Diagnosis: Why Lexicon doesn't remember

User asked why Lexicon responded "Unable to summarize — no prior context" despite having a full day's work logged.

### Root cause — two compounding issues

**1. System loads context correctly, agent doesn't use it:**
- `Registry.loadAll()` reads daily logs fresh from disk after every task (via `refreshTeammates()`)
- `buildTeammatePrompt()` injects today's daily log at priority 100 (never trimmed)
- Lexicon's prompt DID contain its daily log with all prior work
- The agent (Claude running as Lexicon) didn't reliably reference the context in its prompt

**2. Lexicon has almost no memory to work with:**
- WISDOM.md is empty ("No entries yet")
- Recall index has only 2 entries
- Brand new teammate created today — no accumulated context
- Compare to Beacon/Scribe with weeks of dense wisdom and hundreds of recall entries

### Contributing factor — no structural anchors
The prompt uses `##` markdown headers with no structural tags. Daily log content is indistinguishable from other markdown in the prompt. Lexicon's own section-tag spec (`section-tag-spec.md`) diagnoses exactly this problem — but it was never implemented.

### Recommended fixes
1. Implement Lexicon's section-tag + reorder specs in `adapter.ts`
2. Compact Lexicon's daily log into WISDOM.md
