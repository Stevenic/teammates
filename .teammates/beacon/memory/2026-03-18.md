## Task: Diagnose empty teammate responses

User reported constant empty responses from teammates during @everyone brainstorm sessions. Only Scribe responded to a "pitch your favorite idea" prompt; Pipeline, Beacon, and Forge all returned empty.

### Root cause analysis

The prompt sent to each teammate has **5 layers of size explosion**:

1. **Conversation history stores rawOutput verbatim** — `storeResult()` at cli.ts:90-93 stores `result.rawOutput ?? result.summary`. Every teammate's full response (including markdown tables, code blocks, etc.) goes into the history array.

2. **No token budget on conversation context** — `buildConversationContext()` at cli.ts:96-105 has only a 10-entry count limit. With @everyone dispatching to 4 teammates per round + user messages, 10 entries can easily be 50k+ tokens of accumulated brainstorm output.

3. **Conversation context is prepended to the task** — orchestrator.ts:103-104 puts `extraContext` before the task, so the actual instruction ("pitch your favorite idea") gets buried after pages of history.

4. **Base prompt overhead is already large** — SOUL.md (~145 lines) + WISDOM.md + recall results + 7 daily logs (Beacon's 03-17 log alone is 678 lines!) + weekly summaries + session history + roster + session state + memory update instructions + output protocol = thousands of tokens before any task or history.

5. **@everyone amplifies everything** — Each teammate receives the same bloated history, but the agent running as Pipeline/Forge has less context budget than Claude running as Scribe (different agent backends may have different limits).

### Estimated prompt sizes for the failing session

- Base prompt (SOUL + WISDOM + daily logs + instructions): ~3,000-8,000 tokens per teammate
- Conversation history (4 brainstorm responses + consolidation + voting rounds): ~30,000-50,000 tokens
- Actual task: ~20 tokens ("pitch your favorite idea")

The task is <0.1% of the prompt. Agents likely hit context limits or the task gets lost.

### Proposed fixes (3 tiers)

**Tier 1 — Immediate (fix the bleeding):**
1. `storeResult()` should store `result.summary` (the `# Subject` line), not `rawOutput`
2. Cap each conversation history entry to ~500 chars with truncation marker
3. Add a total token budget (~4,000 tokens) to `buildConversationContext()` — trim oldest entries first

**Tier 2 — Prompt diet (reduce overhead):**
4. Condense Memory Updates section — it's 12 lines of instructions that repeat every task
5. Condense Output Protocol section — currently ~20 lines, could be 5
6. Limit daily log injection based on token budget (currently injects 7 days unconditionally)
7. Cap recall results to a token budget, not just a count

**Tier 3 — Structural (longer term):**
8. Conversation context should go AFTER the task, not before — the agent reads the task first, then has history for reference
9. For @everyone tasks, skip conversation history entirely (each teammate gets a fresh prompt with just the task)
10. Implement proper summarization — when history exceeds budget, summarize older entries instead of truncating

### Key decisions
- Diagnosis only, no code changes yet — waiting for user input on which fixes to prioritize
- The core issue is architectural: unbounded conversation history + large base prompt = context explosion

- None (diagnosis only)

## Task: Implement context window budget management

User confirmed the prompt bloat diagnosis and asked for two specific changes: use summary for conversation history, and add a ~64k token budget for the prompt wrapper.

### What was done

1. **`storeResult()` in cli.ts** — Changed from `result.rawOutput ?? result.summary` to just `result.summary`. Each conversation history entry is now ~1 line instead of the full agent output (which could be thousands of lines of tool calls, file edits, etc.).

2. **`buildConversationContext()` in cli.ts** — Added a 16k char budget (~4k tokens). Builds entries from newest to oldest, stops when budget is exceeded. This ensures recent conversation is always available while old entries get dropped gracefully.

3. **`buildTeammatePrompt()` in adapter.ts** — Full refactor to a priority-based budget system:
   - Each prompt section gets an `order` (position in prompt) and `priority` (importance for trimming)
   - Wrapper sections have a 64k token budget (256k chars). Task prompt is excluded from the budget.
   - When over budget, lowest-priority sections are dropped first:
     - Weekly summaries (priority 10) → Session content (20) → Older daily logs days 4-7 (25) → Recent daily logs days 1-3 (35) → Recall results (40)
     - Identity, wisdom, roster, services, handoff, protocol, memory instructions = priority 100, never trimmed
   - Daily logs split into two tiers for progressive trimming: recent 3 days vs older 4 days

4. **3 new tests** — Verify budget trimming drops low-priority sections, keeps all when under budget, and excludes task from budget.

### Key decisions
- `result.summary` is the `# Subject` line from the output protocol — short and descriptive
- Token estimation: 4 chars ≈ 1 token (simple, good enough)
- Task prompt excluded from budget per user: "if the user pastes 32k tokens that's fine"
- Progressive daily log trimming (2 tiers) is more graceful than all-or-nothing

## Task: Bottom-up budget allocation with daily log priority

User feedback: daily logs should always be prioritized over weekly summaries because weekly summaries are already indexed by recall (redundant in the prompt). Replaced the per-section truncation budget system with bottom-up whole-entry allocation.

### What was done

1. **Bottom-up allocation** — Replaced `SECTION_BUDGETS` + `truncateToTokenBudget()` with a linear bottom-up allocator. Each tier adds whole entries until budget is consumed or entries exhausted. Unused budget flows upward to the next tier.

2. **New allocation order** (32k token context budget):
   - Tier 1: Older daily logs (days 4-7) — fill first
   - Tier 2: Previous daily logs (days 2-3) — fill next
   - Tier 3: Recall results — gets remaining budget (query-matched, highest signal per token)
   - Tier 4: Weekly summaries — only if budget remains after recall (already indexed by recall)

3. **Simplified prompt assembly** — Removed `PromptSection` type, priority-based assembly, and global budget safety net. The function now builds a `parts[]` array linearly. All non-context sections (identity, wisdom, roster, protocol, task) are always included. Context sections go through the bottom-up allocator.

4. **Whole entries only** — No more mid-entry truncation. An entry either fits entirely or is skipped. This keeps context coherent — no partially cut-off daily logs.

### Key decisions
- Daily logs fill before recall because recall results are already semantically filtered (small, high-signal entries that usually fit)
- Weekly summaries are dead last because recall already indexes them — they're redundant
- `estimateTokens()` replaces `truncateToTokenBudget()` — simpler, just counts chars/4
- Current daily log (today) remains outside the budget — always included

## Task: Conversation history budget with auto-summarization

User requested a structured context window budget: fixed sections always included, recall context up to 32k tokens, conversation history with a 24k token minimum for recent messages plus a running summary of older history maintained by the coding agent.

### What was done

1. **`conversationSummary` property** — New string property on `TeammatesREPL` tracks the running summary of older conversation history. Reset on `/clear`.

2. **`CONV_HISTORY_CHARS = 96_000`** (24k tokens) — Replaced the old 16k char budget with a 24k token budget for recent conversation history.

3. **`buildConversationContext()` rewrite** — Now produces two subsections: "Previous Conversation Summary" (the running summary of older history) and the most recent entries that fit within 24k tokens. Both are included under `## Conversation History`.

4. **`maybeQueueSummarization()`** — Called after every `storeResult()`. Counts total tokens of conversation history. If over 24k, identifies the oldest entries that don't fit, builds a summarization prompt (including the existing summary if present), removes the old entries from `conversationHistory`, and queues a `"summarize"` task to the base coding agent.

5. **`"summarize"` queue entry type** — New variant in the `QueueEntry` union. In `drainAgentQueue()`: skips conversation context (it's about the conversation itself), skips `storeResult()`, skips `writeDebugEntry()`. Captures `result.rawOutput`, strips protocol artifacts, stores as `conversationSummary`.

6. **Display suppression** — `handleEvent` checks `agentActive` for `"summarize"` type and breaks before any display logic. Summarization is invisible to the user.

7. **Summarization prompt** — Tells the agent to return ONLY a concise bulleted summary (under 2000 chars), no output protocol. Handles both first-time creation and incremental update of existing summary.

### Overall context window budget
| Section | Budget | Behavior |
|---------|--------|----------|
| Fixed (identity, wisdom, roster, protocol, today's log) | Unlimited | Always included |
| Daily logs (days 2-7) | 24k tokens | Whole entries |
| Recall results | 8k+ tokens | Minimum 8k + unused daily budget |
| Conversation history (recent) | 24k tokens | Most recent whole entries |
| Conversation summary (older) | ~500 tokens | Agent-maintained running summary |
| **Total** | **~64k tokens** | Slightly over is fine |

### Key decisions
- Summarization is an async background task — doesn't block the main conversation flow
- Older entries are spliced from `conversationHistory` after being queued for summarization (they'll live in the summary)
- The summarization prompt explicitly says NO output protocol — but we strip it anyway in case the agent includes it
- `kickDrain()` is called after queueing the summarize task so it runs promptly
