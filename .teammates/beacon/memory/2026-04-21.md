## Task: Per-tab task queues (same teammate in two tabs concurrently)

Changed serialization from "one task per teammate" to "one task per (thread, teammate) slot". Same teammate in two different tabs now runs concurrently instead of queueing.

### Why
User asked: "each tab should have its own task queue. The common status bar should rotate through all active tasks but I should be able have the same teammate working on two different things in two tabs."

### Design — the "slot" key

A slot is a `(threadId, teammate)` pair. Helper added: `slotKey(threadId, teammate) => "t{threadId ?? 0}:{teammate}"`. Three core maps rekeyed:

- `agentActive: Map<string, QueueEntry>` — keyed by slot (was by teammate)
- `abortControllers: Map<string, AbortController>` — keyed by slot
- `agentDrainLocks: Map<string, Promise<void>>` — keyed by slot

`isAgentBusy(teammate)` → `isSlotBusy(threadId, teammate)`. `drainAgentQueue(agent)` → `drainSlot(threadId, teammate)`. `kickDrain()` now iterates unique (threadId, teammate) pairs instead of unique teammates.

### StatusTracker — driven from drain loop now

Previously driven from `handleEvent` (orchestrator events), keyed by `teammate`. That breaks because two concurrent tasks for the same teammate would have the same StatusTracker ID — second `startTask` would clobber the first, `stopTask(teammate)` would kill both.

Moved StatusTracker lifecycle into `drainSlot`:
- `startTask(entry.id, ...)` at loop start — unique per task
- `stopTask(entry.id)` in finally at loop end
- `handleEvent` now only handles the `error` case for display

StatusTracker already rotates through all active task IDs in its animation — no change needed there. Status bar naturally shows both tabs' tasks.

### Activity manager — rekeyed by taskId (entry.id)

Activity buffers/shown/lineIndices/threadIds/blankIdx were all keyed by teammate. Two concurrent tasks for @beacon would collide in the buffer, render into each other's containers, etc. Rekeyed all to `taskId` (the queue entry ID). `initForTask`, `handleActivityEvents`, `cleanupActivityLines`, `insertActivityHeader`, `insertActivityLines`, `rerenderActivityLines`, `toggleActivity` — all parameters renamed from `teammate` to `taskId`. The cli.ts onActivity callback now passes `taskId` instead of `teammate`.

### Adapter/orchestrator — no changes needed

- Adapters spawn a fresh child process per `executeTask()` call. Two tasks = two processes.
- `orchestrator.sessions: Map<teammate, sessionId>` — session IDs are synthetic tokens; `cli-proxy.executeTask` ignores the session ID anyway. Safe to share.
- `startSession` uses an incrementing counter — idempotent enough.
