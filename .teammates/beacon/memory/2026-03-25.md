## Task: Fix system tasks blocking user task progress bar

User reported actual task progress not showing after the system task filtering changes.

### Root cause
`silentAgents` was agent-level, not task-level. When `runSystemTask()` added a teammate to `silentAgents` for wisdom distillation, it suppressed ALL events for that agent — including user task `task_assigned` events. During startup, all teammates got wisdom distillation queued simultaneously, putting them all in `silentAgents`. Any user task assigned during that window (~30s-2min per agent) had its progress bar completely suppressed.

### What was done
- Added `system?: boolean` to `TaskAssignment` and `TaskResult` in types.ts
- `runSystemTask()` now passes `system: true` to `orchestrator.assign()` for both summarize and system agent tasks — removed `silentAgents` usage entirely
- `orchestrator.ts` propagates `system` flag from assignment to result
- `handleEvent` checks `event.assignment.system` / `event.result.system` — skips `activeTasks` management for system tasks only, user task events flow normally even for the same agent
- Removed `startupMaintenance` setProgress calls (system-level messages, shouldn't be in progress bar)
- Guarded `runCompact` error handler's `setProgress(null)` with `!silent`
- `silentAgents` retained only for the defensive retry in `drainAgentQueue`

### Key decisions
- Task-level filtering (via `system` flag on events) replaces agent-level filtering (`silentAgents`) for system tasks
- This allows system and user tasks for the same agent to run concurrently without interference
- `silentAgents` still used for defensive retry (that's a different pattern — temporary, per-drain-loop)

### Build & test
- TypeScript compiles cleanly (strict mode)
- 224 CLI tests pass

### Files changed
- `packages/cli/src/types.ts` — `system?: boolean` on TaskAssignment and TaskResult
- `packages/cli/src/orchestrator.ts` — propagate system flag to result
- `packages/cli/src/cli.ts` — `handleEvent` system task filtering, `runSystemTask` uses system flag instead of silentAgents, removed startupMaintenance progress calls, guarded runCompact error setProgress

## Task: Fix conversation history — store full message bodies, not just subjects

User reported that conversation history passed to agents contained only subject lines, not message bodies. Agents couldn't reference prior conversation content (votes, brainstorms, etc.) because the data was stripped.

### Root cause
`storeResult()` in cli.ts was storing `result.summary` (the `# Subject` header from the agent's output protocol) into `conversationHistory`. When `buildConversationContext()` rendered the history for the next agent's prompt, it produced lines like `**scribe:** Scribe's Pick + Votes` — subject only, no body.

### What was done
- `storeResult()` now stores the full cleaned `rawOutput` body (protocol artifacts stripped) instead of just `result.summary`
- Falls back to `result.summary` if `rawOutput` is empty
- `buildConversationContext()` formats multi-line entries with body on the next line for readability
- `maybeQueueSummarization()` updated to match the same multi-line format
- Existing 24k token budget + auto-summarization system handles the increased size naturally

### Key decisions
- Same protocol-stripping regex as `displayTaskResult` (TO: header, handoff blocks, trailing JSON)
- The `# Subject` header is intentionally NOT stripped from stored text — it provides a useful label at the top of each entry
- Multi-line detection via `text.includes("\n")` — user messages stay inline, agent responses get block format

- 224 CLI tests pass
