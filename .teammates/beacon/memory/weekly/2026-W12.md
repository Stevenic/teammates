## Task: Ctrl+click to open web links in browser

Added ctrl+click URL detection to the ChatView widget (consolonia) and wired it up in the CLI to open links in the default browser.

### What was done
1. **ChatView** (`packages/consolonia/src/widgets/chat-view.ts`):
   - Added `URL_REGEX` constant for `https?://` URL detection
   - Added `_screenToFeedRow` map (tracks which row within a multi-row wrapped line each screen Y corresponds to)
   - Cached `_feedX` and `_contentWidth` from render pass for position math
   - Added `_extractFeedLineText()` helper to get plain text from a feed line (handles both plain strings and StyledSpan segments)
   - Added `_findUrlAtOffset()` helper for precise URL hit-testing at a character position
   - Added ctrl+left-click handler: detects URLs in the clicked feed line, resolves which URL based on click position (accounts for line wrapping), emits `"link"` event with the URL
2. **CLI** (`packages/cli/src/cli.ts`):
   - Added `"link"` event listener that opens URLs via platform-appropriate command (`open` on macOS, `start` on Windows, `xdg-open` on Linux)

### Key decisions
- Ctrl+click is checked before regular action click — so ctrl+clicking an action line with a URL opens the URL, not the action
- For single-URL lines, opens immediately without position math
- For multi-URL lines, calculates approximate character offset from click position (row * contentWidth + col) and finds the URL at that offset
- Falls back to first URL on the line if position math doesn't hit exactly

## Task: Diagnose empty teammate responses

User reported constant empty responses from teammates during @everyone brainstorm sessions. Only Scribe responded to a "pitch your favorite idea" prompt; Pipeline, Beacon, and Forge all returned empty.

### Root cause analysis

The prompt sent to each teammate has **5 layers of size explosion**:

1. **Conversation history stores rawOutput verbatim** — `storeResult()` at cli.ts:90-93 stores `result.rawOutput ?? result.summary`. Every teammate's full response (including markdown tables, code blocks, etc.) goes into the history array.

2. **No token budget on conversation context** — `buildConversationContext()` at cli.ts:96-105 has only a 10-entry count limit. With @everyone dispatching to 4 teammates per round + user messages, 10 entries can easily be 50k+ tokens of accumulated brainstorm output.

3. **Conversation context is prepended to the task** — orchestrator.ts:103-104 puts `extraContext` before the task, so the actual instruction ("pitch your favorite idea") gets buried after pages of history.

4. **Base prompt overhead is already large** — SOUL.md (~145 lines) + WISDOM.md + recall results + 7 daily logs (Beacon's 03-17 log alone is 678 lines!) + weekly summaries + session history + roster + session state + memory update instructions + output protocol = thousands of tokens before any task or history.

## Diagnosis: Scribe empty responses

User reported @scribe completing tasks twice with "(no response text — the agent may have only performed tool actions)." Investigated both debug logs.

### Root cause
The agent (Claude via `-p` mode) is ending its turn with file edits (memory + session updates) but producing **no text output**. In `-p` mode, only text portions of the agent's response go to stdout — tool-only turns produce empty stdout. The CLI then sees an empty `rawOutput` and shows the "no response text" fallback.

### Evidence from debug logs
Both logs show identical patterns:
1. Agent reads SOUL.md, WISDOM.md, session file, daily log (4-5 Read tool calls)
2. Agent edits daily memory file + session file (2 Edit tool calls)
3. Final API call: stream starts → session ends within seconds (no text produced)

The agent does all its housekeeping (memory/session writes as instructed by "Before returning your result, update your memory files") and then ends the turn without producing the required `TO: user` text response.

### Contributing factors
1. **Instruction ordering** — The prompt says to update memory/session "before returning your result," so the agent does those first. But there's no hard guardrail ensuring it also produces text afterward.
2. **Output protocol buried** — The `TO: user` output protocol is near the end of a very long prompt. After processing all the memory/session instructions, the agent may lose sight of it.
3. **Large prompt overhead** — SOUL.md + WISDOM.md + daily logs + recall + conversation history leaves less room for the agent to plan a proper response.

### Recommended fixes (3 options)
1. **Prompt restructuring** — Repeat the critical output protocol instruction AFTER the memory update section (not just before the task). Add a final line like: "REMINDER: You MUST produce visible text starting with TO: user. A turn with only file edits is a failed turn."
2. **Two-phase execution** — Restructure the prompt to make text response FIRST, memory updates SECOND. This way the agent always produces text before it gets to housekeeping.
3. **Defensive retry** — If rawOutput is empty and exit code is 0, queue a lightweight retry that just asks for the text response without memory updates.

User reported that multiple teammates (Scribe, Lexicon) claim to queue handoffs but nothing actually happens. Investigated debug logs.

### Root cause
Agents are NOT using the ```handoff fenced code block format. They write about handoffs in plain English instead:
- Scribe: "And yes — please kick off the Beacon handoff whenever you're ready" (no block)
- Scribe: "Handoff to Beacon is done." (no block)
- Lexicon: "All done. 18 files updated, 2 handoffs queued" (no block)

The `findHandoffBlocks()` parser requires exact `/```handoff\s*\n@(\w+)\s*\n/` format — correctly finds nothing because there's nothing to find.

### Why agents don't follow the format
The handoff instruction was buried as one paragraph inside a 20-line Output Protocol section. The format example used nested backticks (code fence inside code fence) which was ambiguous.

### What was done

**Fix 1 — Separated handoff protocol into its own section (adapter.ts):**
- Pulled handoff instructions out of Output Protocol into a new `## Handoffs (IMPORTANT)` section
- Used 4-space indentation for the format example instead of nested code fences — unambiguous rendering
- Added explicit warning: "Mentioning a handoff in plain English does NOT work"
- Added "Do NOT just say 'I'll hand this off' in prose — that does nothing"

**Fix 2 — Natural-language handoff fallback (cli-proxy.ts):**
- Added `findNaturalLanguageHandoffs()` as a fallback when no fenced blocks are found
- Detects patterns: "hand off to @name", "handing this to @name", "delegate to @name", "queued handoff to @name"
- Only matches known teammate names (from `teammateNames` list)
- Extracts surrounding text as the task description
- Deduplicates by target name
- Only fires when zero fenced blocks are found (fenced blocks are always preferred)

### Two layers of defense
1. **Better prompt** — agents should follow the fenced block format now that it's clear
2. **Fallback parser** — catches the common failure mode when agents still use natural language

## Diagnosis: Why Lexicon doesn't remember

User asked why Lexicon responded "Unable to summarize — no prior context" despite having a full day's work logged.

### Root cause — two compounding issues

**1. System loads context correctly, agent doesn't use it:**
- `Registry.loadAll()` reads daily logs fresh from disk after every task (via `refreshTeammates()`)
- `buildTeammatePrompt()` injects today's daily log at priority 100 (never trimmed)
- Lexicon's prompt DID contain its daily log with all prior work
- The agent (Claude running as Lexicon) didn't reliably reference the context in its prompt

**2. Lexicon has almost no memory to work with:**
- WISDOM.md is empty ("No entries yet")
- Recall index has only 2 entries
- Brand new teammate created today — no accumulated context
- Compare to Beacon/Scribe with weeks of dense wisdom and hundreds of recall entries

### Contributing factor — no structural anchors
The prompt uses `##` markdown headers with no structural tags. Daily log content is indistinguishable from other markdown in the prompt. Lexicon's own section-tag spec (`section-tag-spec.md`) diagnoses exactly this problem — but it was never implemented.

### Recommended fixes
1. Implement Lexicon's section-tag + reorder specs in `adapter.ts`
2. Compact Lexicon's daily log into WISDOM.md
