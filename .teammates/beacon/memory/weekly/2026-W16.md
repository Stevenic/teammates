## Task: Implement Thread UX v2 (Tab-Style Task Switching)
Implemented the full Thread UX v2 spec from `.teammates/scribe/docs/specs/F-thread-ux-v2.md`. Major architectural refactor replacing the single-shared-feed threading model with per-thread tab-based feeds. Bumped all packages from 0.8.1 to 0.9.0.

### Architecture changes
- **Per-thread FeedStore**: Each thread now owns its own `FeedStore` instance. Cross-thread index shifting is eliminated — the #1 source of complexity in the old model.
- **FeedAdapter**: New `packages/cli/src/feed-adapter.ts` wraps a FeedStore and provides the `ThreadFeedView` interface. Replaces ChatView as the mutation target for thread content.
- **ThreadBar widget**: New `packages/consolonia/src/widgets/thread-bar.ts` — 1-row tab bar with thread names, unread ● badges, working ◎ indicators, [x] close buttons, and [+] new thread button.
- **ChatView docked bar + store swapping**: Added `dockedBar` option for pinned widgets between banner and feed. Added `setStore()` / `store` getter for swapping active FeedStore when switching tabs.

### Behavior changes per spec
- Session starts with thread #1, tab bar hidden (single-thread mode)
- `/thread [description]` creates a new tab and switches to it
- `#id` alone switches focus (no message required)
- `@mention` dispatches within the current thread (no longer creates new threads)
- `/clear` clears only the focused thread's feed
- `/close [#id]` removes a thread tab (thread #1 permanent)
- `/threads` lists all threads with status
- Tab bar appears when >1 thread exists, docks at top

### Files created

## Task: Always show `<` `>` nav arrows, treat `[+]` as a pageable tab
Reworked ThreadBar pagination: both nav arrows are now always rendered and disabled based on position, rather than appearing/disappearing. The `[+]` button is treated as a virtual trailing tab in the pagination model.

### Behavior
- `<` is always visible. Disabled (rendered in `separator` style, no hit region) when `_pageStart === 0`.
- `>` is always visible. Disabled when the visible page reaches the `[+]` sentinel.
- `[+]` participates in the width/fit calculation — when it can fit after the last visible tab, it's drawn inline and `>` becomes disabled.
- At least one real tab is always shown (even when only the `[+]` would fit), so `_pageStart` is clamped to `[0, tabs.length - 1]`.
- Clicking `>` only advances `_pageStart` up to `tabs.length - 1` (the last real tab).

### Implementation
- Replaced the old `hasPrev`/`hasNext` conditional rendering with always-rendered arrows that swap between `add`/`separator` styles.
- Refactored the fit calculation into a single `fitFrom(start)` helper that iterates over `itemCount = tabs.length + 1` items (tabs + `[+]` sentinel).
- Removed the old `ADD_W - 1 gap` reservation from `availWidth` since the sentinel is now a first-class item in the fit loop.
- Kept the focused-tab-overflow backtrack logic, simplified to call `fitFrom()` again after adjusting `_pageStart`.

### Files changed
- `packages/consolonia/src/widgets/thread-bar.ts` — rewrote `render()` pagination, added `fitFrom()` helper, updated nav-arrow rendering, clamped `next` click advance

## Task: Resolve clicked file paths to absolute + underline links/files on hover
Two fixes to the feed-click UX:

1. **Absolute path resolution** — The `file` click handler in cli.ts was passing the matched path straight to `start`/`open`/`xdg-open`. On Windows, Unix-style `/foo/bar` paths are drive-relative (resolve to `C:\foo\bar`) and often don't exist, and any relative path would fail for the same reason. Now the handler always resolves to an absolute path via `resolvePath(process.cwd(), ...)` before invoking the OS opener. Windows drive-letter paths (`C:\foo`) pass through normalized.

2. **Hover underline on URLs and file paths** — ChatView now detects URL/file-path regions under the mouse cursor and underlines just the matched region until the mouse moves off. Implementation:
   - New `_linkTargetAt(x, y)` helper reuses `URL_REGEX`/`FILE_PATH_REGEX` and the ctrl+click charOffset math to identify the exact target.
   - New `_applyUnderlineOverlay(lines, start, length)` walks the item's `content.lines` segment-by-segment and splits segments that intersect the range, applying `{ underline: true }` only to the intersecting char range.
   - Move handler caches the item's original `content.lines`, swaps in the overlay version, invalidates the item, then restores on exit.
   - Items with action entries are skipped — the action hover subsystem owns their content mutation.
   - Cleared alongside `_hoveredItemId` in `setStore`, `clear`, `updateFeedLine`, `updateActionList`.

## Task: Fix `<` / `>` arrows not paging tabs past the focused one
The ThreadBar was forcing the focused tab back into view on every render and every `tabs` setter call. Clicking `>` would advance `_pageStart`, but the render pass immediately snapped it back because the focused tab fell behind the new page. Same in reverse for `<`. Hover highlighting worked because that's a local UI state — but the actual tab list never scrolled.

### Root cause
Three separate blocks unconditionally re-centered the focused tab:
1. `tabs` setter — `if (focusedIdx < this._pageStart) this._pageStart = focusedIdx`
2. `render()` top — same backward clamp
3. `render()` post-fit — pull `_pageStart` forward when focused overflowed `visEnd`

All three ran on every invalidate, so user pagination was overridden instantly.

### Fix
Changed to a one-shot `_scrollToFocused` flag:
- Added `_lastFocusedId: number | null` and `_scrollToFocused: boolean` fields.
- `tabs` setter only sets `_scrollToFocused = true` when the focused tab id actually changes vs. `_lastFocusedId` (then updates `_lastFocusedId`).
- Both render-time focus-clamping blocks are now gated on `this._scrollToFocused`.
- Flag is cleared after being consumed in `render()`.

Net effect: focus changes (user switches tab, new tab created, tab closed) still auto-scroll the focused tab into view — but manual `<` / `>` paging is preserved across subsequent invalidates (e.g. working-state updates).

### Files changed
- `packages/consolonia/src/widgets/thread-bar.ts` — added `_lastFocusedId` and `_scrollToFocused` fields; gated both focus-clamp blocks in `render()` behind the flag; setter sets flag only on focus id change
