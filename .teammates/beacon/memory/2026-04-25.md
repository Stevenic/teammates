## Task: Add `@everyone` no-redundant-handoffs rule to Handoffs section

User feedback: when Steve sent `@everyone one line intros`, beacon delivered the intro correctly but ALSO issued handoffs to @lexicon and @pipeline -- who were already running the same task themselves via the broadcast. Result: same task ran twice for those teammates.

### Root cause
Handoffs section had no rule about broadcast messages. Teammates assumed they needed to forward broadcast tasks to other teammates the same way they would for narrowly-addressed tasks.

### Fix
- `packages/cli/src/system-prompt.ts` -- added new bullet to the `Rules:` list under `### Handoffs` (right after the "I'll hand this off" prose rule). The rule explicitly covers both the direct `@everyone` case AND the case where conversation history shows the same task is being fanned out to multiple teammates. Clarifies that handoffs are for *delegating expertise* (e.g. needing someone to edit a file you don't own), not *forwarding broadcasts*.
- adapter.ts mirror -- handed off to @lexicon (owns adapter.ts) with a condensed version of the rule for the fallback inline builder.

### New rule text (system-prompt.ts version)
> **`@everyone` rule (NO redundant handoffs)** -- If the user's message was addressed to `@everyone`, every teammate already receives it directly and responds independently. Do NOT issue handoffs to other teammates for the same task -- that would make them run it twice. Just deliver YOUR own response in your own voice. The same applies if you can tell from conversation history that the same task is already being fanned out to multiple teammates. Handoffs are for delegating work that requires another teammate's specific expertise (e.g. you need them to edit a file you don't own), not for forwarding a broadcast.

### Files changed

### Why this matters (durable principle)
Three handoff failure modes now covered in prompts:
1. Handoffs in prose without fenced block (existing rule).
2. Handing off work you should do yourself (existing rule -- "do as much as you can BEFORE handing off").
3. Forwarding broadcast tasks to teammates already receiving them (NEW rule).
Handoff intent matters: are you delegating *expertise* you lack, or *redundantly forwarding* a message?

## Task: Find ROOT CAUSE of lexicon's "Task completed / Intro delivered above" failure (8th attempt)

Steve has tried `@everyone create a one line intro` ~8 times. Lexicon (and previously scribe) keep returning subject `# Task completed` with body `Intro delivered above; daily log updated to note the repeat ask`. Beacon, pipeline, scribe were delivering correctly after my prior fixes -- but lexicon STILL failed even with all the new banned-subject/banned-body rules in place.

### Root cause -- found by reading lexicon's actual prompt debug file
Read `.teammates/.tmp/debug/lexicon-2026-04-25T01-40-40-682Z-prompt.md`. The `## Daily Log` section appended at the end of the user message contained lexicon's OWN note from earlier today: "**One-line intro:** Steve asked @everyone for one-line intros. Delivered a single-sentence persona summary..."

**Lexicon was reading its own daily log, seeing "Delivered..." (past tense), and concluding the current task was already done.** It then ack'd out with a meta-status response. The Output Protocol rules I added previously (banned subjects, banned bodies, @everyone teammate rule) did NOT cover this specific failure mode -- a teammate using its OWN daily log as evidence that the current `<TASK>` doesn't need delivery.

Beacon/pipeline/scribe didn't trip on this because their daily logs either lacked past-tense delivery entries or had highly technical entries that didn't suggest "already done".

### Bonus root cause -- doubled `## Conversation History` header
While reading the prompt file I noticed:
```
## Conversation History
## Conversation History

**stevenic:** ...
```
`buildConversationContext()` in `cli-utils.ts` returned a string starting with `## Conversation History\n`, and `buildUserMessage()` in `adapter.ts` wrapped it AGAIN with `## Conversation History\n${conversationHistory}`. Result: doubled header in every prompt that included conversation history.

### Fix 1 -- new "fresh request" rule in Output Protocol
Added a new all-caps paragraph in BOTH prompt builders, immediately after the "daily logs are PRIVATE" paragraph:

> **EACH `<TASK>` IS A FRESH REQUEST. The Daily Log section at the bottom of this prompt is HISTORICAL CONTEXT — it is NOT proof that the current `<TASK>` is already done.** If your daily log says you delivered something earlier today (e.g. `Delivered one-line intro`), that was a PRIOR turn. The user is asking AGAIN now, in THIS turn, and your prior delivery is invisible to them. **You must produce the deliverable AGAIN in this turn's response, in full.** Do not respond with `Already delivered`, `See above`, `Posted earlier`, or any other reference to past work. Reproduce it. Every. Time.

Also expanded the **Banned bodies** list with three new entries: `Already delivered earlier today`, `See daily log`, `Same as before`.

### Fix 2 -- doubled-header bug
Moved header ownership to `buildUserMessage()` (the consumer). `buildConversationContext()` (the producer) now returns just the entries + optional summary, no `## Conversation History` header. This makes the public API more flexible -- library users can construct conversation strings without worrying about pre-pending the section header. Updated the corresponding cli-utils.test.ts assertion (now expects header NOT to be in `buildConversationContext` output).
