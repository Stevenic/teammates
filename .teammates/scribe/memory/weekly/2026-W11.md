## Implemented three-tier memory system (WISDOM)

Major framework overhaul replacing the two-layer memory system (MEMORIES.md + daily logs) with a three-tier system:

- **Tier 1 — Daily Logs** (`memory/YYYY-MM-DD.md`) — unchanged, raw append-only
- **Tier 2 — Typed Memories** (`memory/<type>_<topic>.md`) — individual files with frontmatter, four types: user, feedback, project, reference
- **Tier 3 — WISDOM.md** — distilled principles from compacted memories, read second after SOUL.md

Key decisions:
- Compaction: manual via `/compact` + automatic every 7 days
- WISDOM.md is a single file (not individual files)
- Compacted memories are deleted (not archived)
- Read order: SOUL.md → WISDOM.md → daily logs → memories on-demand
- Wisdom is personal per teammate (no group wisdom)
- Optional INDEX.md for recall support, but not required

## Episodic memory compaction design

User proposed scaling episodic memory: daily→weekly (52 weeks)→monthly (permanent). Designed how this interacts with existing semantic layer (typed memories→wisdom).

Key design:
- Episodic and semantic are orthogonal axes — episodic tracks "what happened when", semantic tracks "what was learned"
- Weekly compaction has dual output: episodic summary + extracted typed memories
- File structure: `memory/weekly/YYYY-Wnn.md`, `memory/monthly/YYYY-MM.md`
- Recall indexes weekly/monthly but not raw dailies
- Wisdom stays independent, fed only by typed memories
- Awaiting user approval before implementing framework changes

Saved full design doc to `.teammates/scribe/docs/EPISODIC-COMPACTION.md` and added pointer in CROSS-TEAM.md.

## Recall integration design for episodic compaction

Expanded the design doc with a full "Recall Integration" section covering:

- **Multi-pass retrieval**: recency pass (last 2-3 weekly summaries, always) + semantic pass (query-driven across all indexed content) + merge/dedup
- **Type-based priority boost**: typed memories score higher than episodic summaries at same similarity — pre-extracted knowledge is more actionable than narrative
- **Frontmatter spec**: weekly/monthly files get `type`, `week`/`month`, `teammate`, `period` fields for recall filtering
- **Prompt assembly flow**: SOUL + WISDOM + 7 dailies + 2 weeklies (direct by adapter) + semantic results (via recall)
- **Recommendation**: adapter.ts loads recent weeklies directly (just file reads), recall handles semantic search on-demand — keeps adapter simple, avoids recall dependency for basic context
- Pushed back on loading "last few days" via recall since adapter already injects last 7 daily logs (covers the full week before weekly compaction)

## Created Memory System documentation with OpenClaw comparison

Created `docs/MEMORY-SYSTEM.md` — comprehensive document covering the full Teammates memory system with a detailed comparison to OpenClaw's memory system at the end.

Document covers:
- The problem (no persistence between sessions)
- Two-axis architecture (episodic + semantic)
- File layout and session startup read order
- All tiers: daily logs, episodic summaries, typed memories, wisdom
- Compaction pipelines (episodic + semantic)
- Recall integration
- Cross-teammate sharing

Comparison with OpenClaw covers:
- File layout differences (3-tier vs 2-layer)
- Memory lifecycle (compaction vs manual curation, automatic flush)
- Search and retrieval (recall vs hybrid BM25+vector with MMR/temporal decay)
- Design philosophy (convention-based vs infrastructure-based)

Key finding: both use plain Markdown as source of truth. Teammates adds semantic structure (types, tiers, compaction). OpenClaw adds search infrastructure (embeddings, indexes, scoring).
