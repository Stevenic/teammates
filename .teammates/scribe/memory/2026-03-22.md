## Code collision prevention — multi-human + multi-agent design
- User asked how to prevent code overwrites when many humans and agents work in the same repo
- Proposed 5-layer defense model:
  1. Branch isolation (each human-agent pair on own branch)
  2. Worktree isolation (S27 — each agent gets its own git worktree)
  3. Ownership routing (PM routes tasks based on SOUL.md file ownership)
  4. Active claims system (NEW — advisory file locks in `.teammates/.tmp/claims/`)
  5. PR merge queue (GitHub-native serialized integration)
- Key insight: textual conflicts (merge conflicts) are easy — semantic conflicts (incompatible refactors that merge cleanly) are the hard problem. Ownership + claims solve this.
- Claims system is the only undesigned piece — file-based, fits Campfire Phase 1
- Offered to write a spec — user approved

## Worktree strategy design — per task with shared home base
- User asked: per agent, per session, or per task?
- Recommended **per task** — maximum isolation, clean lifecycle (create → work → merge/PR → destroy)
- **Key design split:** code changes happen in task worktree, `.teammates/` operations (memory, handoffs, session state) happen in main worktree via absolute path
- This solves the visibility problem: handoffs and memory writes are immediately visible to all agents
- Branch naming: `teammates/<agent>/<task-slug>`
- Claims (Layer 4) stored in `.git/teammates/claims/` — inside shared `.git` dir so all worktrees on same machine can see them
- **When NOT to create worktrees:** memory-only tasks, read-only tasks, `.teammates/`-only changes — detected from teammate's ownership patterns
- Multi-human scenario: each human's agents branch off that human's current branch
- 3 open questions: worktree root location, auto-merge policy, which teammates get worktrees
