## Recall design discussion — chicken-and-egg problem
- User raised fundamental question: recall has a chicken-and-egg problem — you don't know what you don't know
- Three knowledge sources identified: (1) user's current message, (2) conversation history, (3) teammate's own memories
- Currently: adapter passes raw taskPrompt as single embedding search — no query expansion, no conversation context, no memory-informed priming
- Provided design analysis covering 5 strategies: multi-query expansion (LLM generates N queries), conversation-distilled queries, memory-primed search (teammate's memory index as a "what might be relevant" signal), HyDE (hypothetical document embeddings), and two-stage retrieval
- Key insight: the teammate's memory catalog itself is the missing signal — a lightweight keyword/topic index of what the teammate knows could guide query generation without embedding the full corpus

## Recall architecture constraint — two-pass design
- User pointed out the core constraint: query expansion needs an LLM, but recall either needs a small local model (heavy dependency) or the coding agent (which recall can't call — architecture flows one way)
- Proposed two-pass solution:
  1. **Pre-task priming pass (no LLM):** adapter fires raw query at recall, injects results into prompt. Can improve with keyword extraction and memory index scanning — no neural model needed.
  2. **During-task agent pass:** expose recall as a tool (or MCP server) the agent can invoke mid-task. Agent has full context and can craft precise queries.
- This breaks the chicken-and-egg: you don't need a perfect first query, just a good-enough one. The agent self-corrects during execution.
- Plays nicely with S26 (MCP Passthrough) — recall could be an MCP server
- Key architecture principle: recall stays LLM-free (it's a search engine, not a reasoning engine). Heavy reasoning belongs to the agent.

## Human twins collaboration model — design exploration
- User asked to explore how humans participate in the teammates model at scale
- Key framing: each human has a "digital twin" that forms memories around everything you've ever done
- Scenario: 10 human teammates + 5 virtual (AI) teammates
- **Naming decision: "twin" over "avatar"** — twin implies a mirror that learns and can eventually act on your behalf, closer to the digital twin concept from manufacturing/IoT
- **Twin memory formation:** captures everything — git commits, PR reviews, code changes, handoff responses, design decisions, issue comments — not just what happens in the CLI
- **Twin as mediator:** AI teammates talk to the twin first. If the twin can answer from accumulated memory, no human interruption needed. If not, queued for the human.
- **Collaboration at scale:**
  - 5 communication channels: human↔human (twin-mediated), human→AI (direct), AI→human (through twin), AI↔AI (direct handoff), twin↔twin (async context sharing)
  - AI teammates are shared resources — all 10 humans can task the 5 AI teammates
  - PM manages routing, priority, and contention (multiple humans wanting the same AI teammate)
  - Standups need digests at scale — no one reads 15 individual standups
- **Twin as institutional memory:** when a human leaves, their twin stays. New teammates query the departed human's twin for context.
- **Twin maturity model:** passive recorder → context provider → active proxy (mirrors the Phase 1-4 in Campfire)
- Builds on Campfire spec (F-multi-human-collaboration.md) but deepens the twin concept significantly
- User asked for naming feedback — I recommended "twin" over "avatar"
